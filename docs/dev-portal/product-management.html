Product Settings, Sandbox, and Deployment

Overview of managing and configuring your product.

13 mins to read

In Epic Games Store and Epic Online Services (EOS), data relating to your Epic Games account is organized in a hierarchy. Products, sandboxes and deployments are part of the hierarchy.

You manage product, sandbox, and deployment in the Product Settings page.

For an overview of the Epic data hierarchy, see the following documentation:

  • Developer Portal - Introduction - Your Organization and Products

What’s a Product?

A product is a game or other software application that uses one or more EOS service, Epic Games Store distribution, or Epic Games Store service. A product is owned by an organization and contains all information relevant to that software application, such as add-ons, mods, editors, development builds, and distributions on different platforms.

Your organization can have multiple products. You can have multiple versions of a product.

Note: You decide on the names for your products. We recommend setting product names and any associated information to codenames during development. This acts as an additional safeguard for sensitive information. (The default product name comes from the ProductName field in the EOS SDK API EOS_Initialize.)

There is information on sandboxes and deployments, in the Sandboxes and Deployments section of this page.

Product Settings Page

The Developer Portal has a Product Settings page where you can configure data relating to your product, including sandboxes, and deployments.

Page Navigation

Note: In the image above, the left navigation menu also shows other Epic Tools you might choose to use in game development. You do not need these tools to use Epic Games Store for the distribution of your product, either of the sets of Epic Online Services (EOS) - EOS Epic Games Services, EOS Epic Account Services, or to see your product’s analytics.

To access the Product Settings page:

  1. In the Developer Portal (dev.epicgames.com/portal), make sure you are in the correct organization relating to your product. To do this:
    1. Select Organization from the top of the left navigation’s dropdown menu (in the images below, the organization name is “Your…”).
    2. Follow the onscreen instructions to choose your organization.
    3. This displays a list of your organization’s products at the bottom of the left navigation menu under Your Products.
  2. Click your product name from under the list of Your Products in the left navigation menu (in the image below, the product name is “Your Product Name Here”). This displays the following on the left navigation:
    • A list of services your product might use: Epic Games Store, EOS Game Services, EOS Epic Account Services, and Analytics.
    • The Product Settings option.
  3. Click Product Settings. This displays the Product Settings page.

SDK Download & Credentials

In this tab, you can access the EOS SDK downloads and generate credentials so your game can integrate with EOS.

EOS SDK Download

  • For information on the EOS SDK download types, see documentation: Epic Online Services - Get Started Steps: EOS SDK Download Types.

EOS SDK Credentials

These credentials are unique IDs and a client secret (client password) that are specific to your product. Note that there are unique IDs for each tier of the product hierarchy: your product, sandboxes, deployments, clients, and applications.
For details of these terms in the product hierarchy, see the following documentation: Developer Portal - Introduction.

You use these IDs and the client secret in your product’s code when you initialize the SDK with the EOS Platform Interface. You add them in a header file.
For information, see documentation: EOS Game Services - Platform Interface - Creating the Platform Interface.

You can use this tab to generate credentials code that you need to add to your product’s header file. The code is a struct that you can use in either C or C++.

How to Generate Credentials Code
  1. Go to your product’s Product Settings page in the Developer Portal. To do this
    • In the Developer Portal (dev.epicgames.com/portal), from the left navigation, under Your Product, click the name of your product.
    • Once you are in your product page, from the left navigation, click Product Settings.
  2. The default tab is SDK Download & Credentials. In this tab, click Use credentials in header file.
    This displays the Credentials header file window.
    • Note: The code this window generates is a struct that works in either C or C++.
  3. In the window, select the deployment, client, and application you want to generate the code for.
    • Note: To add a client and application, you must agree to the Epic Account Services terms and conditions, and set up at least one client and at least one application. See How to Add a Client and Application section below.
  4. After you have selected the deployment, client, and application in the window, click I understand to acknowledge the warning and generate the code.

For information on client credentials, see the following documentation:

  • Developer Portal - Client Credential and Client Policy Management
  • EOS SDK API Reference - EOS_Platform_ClientCredenti
How to Add a Client and Application
  1. If you haven't already done this, agree to the Epic Account Services terms and conditions in the Agreements tab of the Organizations page.
    To do this:
    1. In the Developer Portal (dev.epicgames.com/portal), click Organization from the left navigation. (Note: You might have to click Back to organization before you can do this.)
    2. On the Organization page, click Agreements and follow the onscreen instructions to review the Epic Account Services agreement.

BPT Credentials

If you are distributing your product on the Epic Games Store, the BuildPatch Tool (BPT) is a command-line tool you use to upload your product binaries to the store. To use the tool successfully, you need to authenticate using the BPT credentials in this tab.

For information about the BuildPatch Tool, see the documentation: Developer Portal - BuildPatch Tool Instructions.

Sandboxes

Use this tab to manage sandboxes and deployments. Because deployments are associated with a sandbox, you create, edit, or archive the deployments in the Sandbox tab of the Developer Portal’s Product Settings page.

For more information, see the Sandboxes and Deployments section of this page.

Identity Providers

Identity providers are real-world organizations providing user accounts, such as a Google account, a Steam account, an Xbox Live account, or Epic Games account, among others.

You can use Epic Online Services (EOS) crossplay to enable your product users to interact regardless of the identity provider they use, and regardless of the storefront or platform they use to access your product.

Using a game as an example, this enables players with accounts from different storefronts or platforms to play your game together. For example, the following players can play your game together:

  • A player on a PC using their Epic Games account
  • A player on a PC using their Steam account.
  • A player on a PlayStation using their PSN account.
  • A player on an Xbox using their Xbox Live account.
  • And players using many other account types and platforms.

With crossplay, players can share their in-game unlocked achievements, see each other on friends lists, and be in sessions and lobbies with each other.

For details of supported identity providers and how to configure them see the documentation: Developer Portal - Identity Provider Management.

Clients

Use this tab to configure your product’s clients and manage your product’s client policies.

What are clients?

Clients are any software program that accesses EOS services for your product. A client can be:

  • User clients:
    • Game clients used by players on their local devices to play your game.
    • Software clients used by software users on their local devices to run your software application.
  • Trusted servers as clients:
    • Your product might use non-EOS services that are complementary to EOS services. These non-EOS services are known as “trusted servers”. Trusted servers require access to your product's user data, so you must add them as a "client".
  • Any other use for a client you choose.

Documentation:

  • Developer Portal - Introduction - Your Organization and Products - an overview of the Epic Games Store and EOS data hierarchy and how clients fit into it.

Note: You must associate a client with client policies. You cannot create a client without a client policy in place.

For further information, see documentation: Developer Portal - Client Credentials and Client Policy Management.

Player Groups

Use player groups to grant users who are not members of your organization, access to your private sandboxes and deployments. By default, private sandbox or deployment access is denied to users who are not members of your organization, unless you give them permission by using player groups.

For example, you can add your product’s QA team to a player group so the team can test a private version of your product.

Warnings:

  • Using Epic Account Services
    If your product is using Epic Account Services, but you have not yet completed the brand review process, user access is denied to non-organization members, even if you have granted them access through player groups.
    For information on the process, see the documentation: Brand Review Process.
  • Members have automatic access
    Members of an organization have user access to their organization’s private sandboxes by default.
    For information on organization members and roles, see the documentation: Developer Portal - Organization - Page Tabs.

Create a New Player Group

To manage deployments:

  1. Go to the Player Groups tab in your product’s Product Settings page in the Developer Portal.

    To do this

    • In the Developer Portal (dev.epicgames.com/portal), from the left navigation, under Your Product, click the name of your product.
    • Once you are in your product page, from the left navigation, click Product Settings.
    • The default tab is SDK Download & Credentials. Click Player Groups.

  2. In the Player Group tab, click Create new group.

  3. Follow the on-screen instructions to create a new group.
    Notes:

    1. Account Ids are unique IDs for user accounts provided by a real-world organization listed in the Identity Provider dropdown list.
    2. Identity Provider is a dropdown list of real-world organizations providing user accounts, such as a Google account, a Steam account, an Xbox Live account, or an Epic Games account, among others. You can only add account IDs to player groups if the account IDs are provided by these identity providers.
      For more information, see the Identity Providers section of this page.

  4. To edit or remove an existing group, click the three dots ( … ) and select the action you want to take.

Note: The Developer Portal uses the terms “user group” and “player group” interchangeably.

Once you finish creating player groups, you can select the sandboxes and deployments this group can access. Once you create the group and give it access to a private sandbox, all users in that player group can access deployments in that private sandbox.

Sandboxes and Deployments

Sandboxes

What is a Sandbox?

A sandbox is an environment for versions of a product. You can use sandboxes to keep the different stages of your product's development and release cycle separate.

As well as a public sandbox (“Live”), for deployments of your live product accessible to users, you can have private sandboxes (“Dev”, “Stage”). You can use private sandboxes to ensure that your product’s deployments for development and testing are not visible to people outside your organization.

A sandbox contains Epic Games Store-related information, mod configurations, and one or more deployments of a game.

Documentation:

  • Developer Portal - Introduction - Your Organization and Products - an overview of the Epic Games Store and EOS data hierarchy and how sandboxes fit into it.

When you set up a product in the Developer Portal, by default, there is one public sandbox called “Live”. You can get two additional sandboxes, “Dev” and “Stage” that default to private to your organization.
Typically you use “Dev” for development deployments and “Stage” for QA testing deployments.

Get Additional Sandboxes

To get the two additional default sandboxes, apply for either of the following:

  • Distribute your product on the Epic Games Store.
    Documentation: Publishing Tools.
  • Use Epic Games Store achievements.
    Documentation: Epic Games Store Achievements.

To get additional sandboxes for a different use, for example to test identity providers (related to using the Auth Interface and Connect Interface):

Contact Epic Games Support using a private discussion.
See the support page section: Using the EOS Developer Support Website

Configure Sandboxes

For each sandbox, you can configure the following:

  • Deployments.
  • Identity providers. (Documentation: Identity Provider Management.)

Deployments

What is a Deployment?

A deployment is an iteration of your game that contains a unique set of player progression data, such as achievements, statistics, and matchmaking. This player progression data includes achievement statuses, statistics, and any ongoing matches in matchmaking.

Documentation:

  • Developer Portal - Introduction - Your Organization and Products - an overview of the Epic Games Store and EOS data hierarchy and how deployments fit into it.

Note: You can associate a deployment with only one sandbox. If you want parallel deployments in two different sandboxes, you must create one deployment for each sandbox.

Manage Deployments

Because deployments are associated with a sandbox, you create, edit, or archive the deployments in the Sandbox tab of the Developer Portal’s Product Settings page.

To manage deployments:

  1. Go to your product’s Product Settings page in the Developer Portal and click Sandboxes.
    To do this

    • In the Developer Portal (dev.epicgames.com/portal), from the left navigation, under Your Product, click the name of your product.
    • Once you are in your products page, from the left navigation, click Product Settings.
    • The default tab is SDK Download & Credentials. Click Sandboxes.

  2. In the Sandboxes tab, click Deployments.

  3. To add a new deployment, In the sandbox deployment’s pane, click New deployment.

  4. To edit or archive an existing deployment, click the three dots ( … ) and select the action you want to take.

Note: If you archive a deployment, neither you nor Epic Games can unarchive it.

Deployment IDs

At creation, each new deployment is automatically assigned a unique identifier.

You need this to integrate your game with EOS, and the Epic Games Store’s Epic Games Launcher.

  • Integrate your game with EOS
    You use the deployment ID to initialize the EOS SDK. This enables EOS services to access product session data for the deployment, as well as information stored at the sandbox level.
    Documentation:

    • EOS Get Started Steps: Step 3 - Integrate the Platform Interface
      • EOS Platform Interface: Initializing the SDK

  • Map to Epic Games Launcher
    When you distribute your product through the Epic Games Store, the Epic Games Launcher uses the deployment ID to map your deployment to the Epic Games Launcher as an artifact.
    Documentation:

  • Epic Games Store: Launcher Invites

Deployment Use-Case Examples

Deployments can provide another level of arrangement and separation to your development, testing, and shipping process. The following examples highlight different deployment approaches.

Single-Branch Deployment
  • Create a single deployment called Live1 for development, testing, and shipping.
    • All the data from development and testing is present in Live1
  • Create two deployments: Dev1 and Live1. Conduct development and testing in the Dev1 deployment, but only ship Live1.
    • All data from development and testing persists in Dev1, but is not present in Live1.
Single-Branch Beta Test Deployment
  • Create two deployments: DevA and LiveA. Conduct development and internal testing in the DevA deployment, and run the beta tests in the LiveA deployment. Then, ship the game using the LiveA deployment.
    • All data from development and internal testing persists in DevA and is not present in the betas or LiveA.
    • All data from the beta tests are present in LiveA.
  • (With wipe) Create three deployments: DevA, BetaA, and LiveA. Conduct development and internal testing in the DevA deployment and run the beta tests in the Beta deployment. Then, ship the game using the LiveA deployment.
    • All data from development and internal testing persists in DevA and is not present in BetaA or LiveA.
    • All data from BetaA persists but is not present in LiveA.
Separate Development Branches

For example:

  • Dev-Main
  • Dev-Release
  • Dev-Experimental
  • Dev-CoolFeature1
  • Dev-CoolFeature2
A CI/CD Pipeline

For example:

  • Dev-Latest
  • Dev-Stable
  • Dev-Staging
  • Live
Separate Development Needs

For example:

  • Dev-LongTermStableBuildForExternalPartner
  • Dev-MainBranchPlaytest
  • Dev-ReleaseBranchPlaytest
